3 approcci architetturali

Come strutturiamo il progetto? Considera complessità, manutenibilità e velocità di sviluppo MVP.

A

Vite + React + tutto in-house

Stack puro: React Router per gli step, Context API o Zustand per lo stato, nessun meta-framework. Massimo controllo, zero magia nascosta. Build statica deployabile ovunque (Netlify, S3, nginx).

Pro

  • Semplicissimo da capire e modificare
  • Bundle piccolo
  • Deploy triviale
  • Nessun lock-in

Contro

  • Routing manuale degli step
  • SSR non disponibile (non serve per MVP)

⭐ Raccomandato per MVP

B

Next.js App Router

Meta-framework full-stack. Ogni step come route `/configuratore/[step]`. Server Components per dati, Client Components per interattività. Pronto per integrare backend Movia in futuro con Server Actions o API Routes.

Pro

  • Route-based step navigation (URL shareable)
  • Facile aggiungere API routes mock
  • Upgrade path naturale verso backend reale

Contro

  • Più complessità per MVP puro frontend
  • Attenzione client/server boundary con Three.js
  • Build più lenta
C

Vite + React + MSW (Mock Service Worker)

Come A ma con MSW per simulare le API Movia in modo realistico (intercetta fetch nel browser). Ogni chiamata API è già cablata — quando il backend è pronto, si disabilita MSW e funziona tutto.

Pro

  • API mock realistiche (non hardcoded)
  • Transizione backend reale senza refactor
  • Dev experience identica a produzione

Contro

  • Setup MSW leggermente più complesso
  • Service worker richiede HTTPS in alcuni contesti

⭐ Alternativa valida se backend Movia è imminente
